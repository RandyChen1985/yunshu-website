很多企业在推进 AI 应用时，会遇到一个很现实的问题：模型能力越来越强，但真正落到业务里，仍然绕不开数据、权限、工具、系统集成和日常运营。

业务人员希望用自然语言查数、问知识、跑流程；技术团队希望数据访问可审计、SQL 可控、接口可治理；运营人员又希望 Agent 能接入微信、飞书、Telegram 等渠道，并且可以稳定地配置、监控和维护。

如果这些能力分散在不同系统里，AI 很容易停留在“能聊”的阶段，很难进入“能用、可管、可持续演进”的阶段。

这也是我做 云枢产品矩阵 的出发点：把企业数据、AI 智能体执行和 Agent 运营入口放在同一条产品线上，让数据可以被治理，让智能体可以执行，让 Agent 可以被运营。

01

云枢不是三个孤立项目，而是一条产品链路

云枢产品矩阵目前包含三层：云枢 · 数据服务平台、云枢 · 智能体平台、OpenClaw Buddy。它们分别覆盖企业智能化链路中的数据供给、智能执行和运营入口。

01

云枢 · 数据服务平台

把物理表、SQL、元数据和 API 治理起来，让数据成为可发现、可授权、可审计的数据产品。

02

云枢 · 智能体平台

承接对话、查数、知识、MCP 工具调用和业务执行，让 AI 从聊天进入真实工作流。

03

OpenClaw Buddy

把开放式 Agent 网关变成非技术人员也能配置、协作、观察和运营的 Web 工作台。

云枢想做的不是“再做一个聊天框”，而是把 AI 真正接到企业的数据资产、权限体系、业务系统和运营入口里。

02

云枢 · 数据服务平台：让数据先变成可运营资产

企业里很多数据并不是不能用，而是“不好被安全、稳定、低成本地使用”。数据可能散落在 MySQL、ClickHouse、Oracle、SQL Server、业务库和数仓里；字段含义依赖口口相传；临时 SQL 越写越多；接口权限、调用审计和生命周期管理也经常缺位。

云枢 · 数据服务平台 把物理表、自定义 SQL 与语义元数据封装为可治理、可审计、可观测的数据产品和 RESTful API，让数据不只是数据库里的表，而是可以被发现、申请、授权、调用和持续运营的产品。

  • 资源服务化：TABLE 零代码映射与 SQL 模板封装
  • SQL Lab：在线编写、调试、AI 辅助生成与修复 SQL
  • 元数据治理：统一管理数据源、表、字段、指标和关系
  • 数据产品目录：按业务域浏览、申请、审批和运营数据产品
  • 安全审计：API Key、Session、RBAC、数据脱敏与 SQL 护栏
  • 全链路观测：调用趋势、Top 排行、分钟级统计和健康状态
03

云枢 · 智能体平台：把 AI 从聊天推进到业务执行

有了数据底座之后，下一步是让 AI 能理解问题、选择能力、调用工具，并把结果以可追踪的方式返回给业务用户。这就是 云枢 · 智能体平台 要解决的事情。

它面向企业复杂场景，提供多专家路由、AgentScope ReAct 执行、ChatBI、知识库、MCP 工具生态、Embed Chat SDK 和全链路 Trace。

入口接入 管理后台、业务系统、数据门户或嵌入式 SDK 发起对话。
意图路由 自动识别任务，由 Assistant、ChatBI、Knowledge 等专家承接。
工具执行 调用数据查询、知识检索、MCP 工具或 OpenClaw 网关。
过程追踪 Trace 时间线记录执行过程，便于调试、审计和复盘。

其中，企业级 ChatBI 是非常重要的一块能力。它不是简单的 Text-to-SQL Demo，而是围绕真实企业环境补齐了元数据注入、Few-Shot 案例、SQL 自愈、SQL 安全护栏、数据门户和执行追踪。

04

OpenClaw Buddy：让 Agent 真正可配置、可协作、可运营

企业 AI 应用走到后期，另一个问题会变得越来越明显：Agent 不只是开发者命令行里的一个进程，它需要被更多角色配置、使用、观察和维护。

OpenClaw Buddy 是为 OpenClaw 打造的 Web 管理台与对话伴侣。它把 Bot、模型、Skills、渠道、Workspace、Console、自愈和审计放到统一界面里，让 OpenClaw 从命令行工具变成可运营的 Web 工作台。

  • Chat V3 工作台：多会话、文件、Skills、Console 和诊断同屏协作
  • Workspace 文件：按 Bot 感知专属工作区，支持浏览、编辑和上传
  • Console 终端：基于 WebSocket 的远程终端定位到当前工作区
  • Bot / Model 管理：管理 Bot、专家模板、模型渠道和默认模型
  • 渠道与插件：可视化管理 Skills、微信、飞书、Telegram、QQ Bot
  • 守护与自愈：心跳探针、备份回滚、审计大屏、系统指标和告警
05

典型落地场景：从查数到嵌入，再到多渠道运营

BI

自然语言查数

业务人员用自然语言查询指标、拆解趋势，结合元数据、权限、SQL 安全链路和执行 Trace。

API

数据产品化

把表、SQL 与接口封装成可发现、可申请、可审计的数据产品，形成统一服务目录。

RAG

知识与数据增强

沉淀企业文档、语义元数据和业务口径，为智能体提供可信上下文与可追溯引用。

MCP

工具生态接入

通过 MCP 连接企业系统、插件与生产力工具，让智能体具备可控的业务执行能力。

SDK

业务系统嵌入

通过 Embed Chat SDK 接入现有鉴权、租户与业务流程，把智能体嵌进真实工作场景。

OPS

Agent 多渠道运营

通过 OpenClaw Buddy 管理微信、飞书、Telegram 等入口，统一会话、授权、日志和告警。

06

交付思路：先建数据底座，再接智能执行，最后补运营入口

云枢产品矩阵不是要求一次性全部落地。更适合的方式，是按企业当前阶段分步推进。

Step 1

先建数据服务底座

梳理数据源、元数据、API 资源和数据产品目录，让数据访问变得规范可控。

Step 2

再接智能体执行

让 ChatBI、知识问答、MCP 工具调用和业务系统嵌入逐步进入真实场景。

Step 3

最后补齐运营入口

当 Agent 被更多角色和渠道使用时，补齐 Bot 管理、渠道插件、工作区协作和运行守护。

这条路线的核心，是让 AI 应用从“单点体验”走向“企业级系统能力”。不是只看模型回答得漂不漂亮，而是看它能不能接上真实数据，能不能遵守权限，能不能调用工具，能不能被追踪，能不能交给业务和运营团队长期使用。

Connect Data. Orchestrate Intelligence. Operate Agents.

真正有价值的 AI 系统，需要能够连接数据、编排执行、进入业务流程，并且在日常运营中保持可控、可观测、可演进。
